### 4. 프로젝트 안내

- **기본**
    - 스프링부트: 3.5.5, 자바: 21
    - 데이터베이스: MySQL 또는 H2 (*테스트가 가능하도록 더미 데이터 삽입용 `.sql` 파일을 반드시 함께 첨부 !!!!)*
    - GitHub: 제출 시 저장소를 Public으로 전환
    - 프로젝트(애플리케이션) 실행을 위해 필요한 조치(환경 변수, 계정, 초기 데이터 등)는 보고서에 구체적으로 작성
    - **빌드 오류 발생 시, 평가가 불가능**

- **필수 기능**
    - **REST API 기반 설계·구현**
        - HTTP 메서드 준수: `POST`(Create), `GET`(Read), `PUT`(Update), `DELETE`(Delete)
        - 표준 상태코드·응답: 상황에 맞는 상태코드와 통일된 JSON 응답
    - **3계층 구현 (Controller–Service–Repository)**
        - RestController는 요청·검증·응답을, Service는 비즈니스 처리·트랜잭션을, Repository는 데이터 접근을 전담한다. (*단, 프로젝트 요구사항에 따라 로깅·캐싱·권한 확인 등을 추가하더라도 계층 간 역할은 섞지 않는다. )*
        - Controller : `@RestController`를 사용해 Spring MVC 애너테이션을 적절히 활용하여 REST 컨트롤러를 구현
        - Service : 인터페이스와 구현 클래스로 분리하여 설계·구현
    - **JPA**
        - 엔티티 설계·구현
        - 쿼리 구현
            - 프로젝트 요구사항 기반으로 Query Method·JPQL·QueryDSL을 상황별로 적용
            - Query Method/JPQL/QueryDSL 중 택1(이상) 구현 필수
    - **요청·응답은 DTO(Data Transfer Object)**
        - 요청·응답은 항상 DTO를 사용하고, 엔티티는 Service/Repository 내부에서만 사용
    
- **추가 기능**
    - 프로젝트 기능 구현·개선
        - 프로젝트 요구사항 개발을 위한 라이브러리/프레임워크 선정·구현(선정 이유를 보고서에 명확히 기술)
        - 라이브러리/프레임워크 도입이 아니더라도 코드 리팩터링, 쿼리 성능 개선, 트러블슈팅 해결 과정을 수행하고, 그 근거와 결과를 보고서에 명확히 기술
    - 테스트 코드 작성
        - JUnit, AssertJ(검증), 그리고 MockMvc를 사용
        - 서비스/리포지토리 중 **1개 이상**과 컨트롤러 **1개 이상**에 대해 테스트를 구현
    - 팀 프로젝트
        - 타입·파일명·패키지 등 일관성 유지
        - 접미사 통일
            - 컨트롤러, 서비스, 리포지토리, 엔티티, DTO 등
        - 공통 모듈/클래스 재사용
            - 예) @MappedSuperclass, @RestControllerAdvice, ApiResponse, ErrorResponse 등
